# 情感对话与心理评估引擎模块需求分析说明书（核心 LLM）

本文定义“模块四：情感对话与心理评估引擎（核心 LLM 部分）”的需求、技术选型、输入输出协议、记忆与知识库策略、安全护栏与评测指标。该模块可采用 API 调用方式接入大语言模型（不要求自建训练）。

对接协议总则见：[protocols.md](../protocols.md)

---

## 1. 目标与技术指标（赛题约束）

### 1.1 功能目标

- 使用语音识别文本作为主要输入（`PerceptionToLLM.asr.text`）
- 具备上下文感知能力，支持 ≥10 轮连续对话
- 输出非预设、非模板化且自然连贯的对话内容
- 具备追问响应、语义澄清、内容总结能力
- 对话逻辑连贯无脱节，交互稳定无中断/闪退
- 基于心理学知识库与风险评估信号，生成更专业且富共情的陪护对话
- 支持对焦虑倾向、抑郁倾向、双向情感障碍风险等进行风险提示与策略分流（不做临床诊断）

### 1.2 性能与资源指标

- 上下文窗口长度：≥ 8k tokens
- 单样本推理响应时间：≤ 60 秒（以 `constraints.deadline_ms` 约束治理）
- 支持流式输出（建议）：先输出可播报的共情短句，再补充引导与总结，降低“体感等待”

## 2. 技术栈与选型（已确定）

### 2.1 核心技术栈

| 组件 | 技术选型 | 版本/说明 |
|------|----------|-----------|
| LLM框架 | **Transformers** | 本地加载，支持4-bit量化 |
| 基座模型 | **Qwen2.5-14B-Instruct** | 中文能力强，8K上下文 |
| 推理优化 | **4-bit量化 (BitsAndBytes)** | NF4量化，显存约10GB |
| 向量数据库 | ChromaDB | 轻量级，本地文件存储 |
| Embedding | **TF-IDF** | **离线可用，无需在线服务** |
| RAG框架 | 自定义实现 | 基于ChromaDB + TF-IDF |
| 编程语言 | Python 3.10+ | 主开发语言 |

### 2.2 基座大模型（已确定）

**主选方案: Transformers本地加载 Qwen2.5-14B-Instruct**

```
模型配置:
- 模型名称: Qwen/Qwen2.5-14B-Instruct
- 上下文长度: 8K tokens
- 推理框架: Transformers (本地加载)
- 量化方案: BitsAndBytesConfig (NF4 4-bit量化)
- 显存占用: 约10GB (GPU) / 可降级到CPU
- 预期延迟: 首token < 2s, 完整响应 < 30s
- **实现位置**: `services/llm/qwen_engine.py`
```

**4-bit量化配置**:

```python
from transformers import BitsAndBytesConfig
import torch

quantization_config = BitsAndBytesConfig(
    load_in_4bit=True,
    bnb_4bit_compute_dtype=torch.float16,
    bnb_4bit_quant_type="nf4",
    bnb_4bit_use_double_quant=True,
)

model = AutoModelForCausalLM.from_pretrained(
    "Qwen/Qwen2.5-14B-Instruct",
    quantization_config=quantization_config,
    device_map="auto",
    trust_remote_code=True,
    local_files_only=True  # 离线模式
)
```

**备选方案: API调用（如本地资源不足）**

- 适用场景: 本地GPU资源不足时
- 推荐API: 阿里云DashScope (百炼)
- 要求: 支持8K上下文、流式输出

**模型选择对比**:

| 模型 | 参数量 | 显存需求 | 中文能力 | 延迟 | 推荐度 |

| Qwen2.5-14B | 14B | ~10GB(4bit) | ★★★★☆ | 中 | ★★★★★ |

### 2.3 心理学知识库（RAG）

- 选型: ChromaDB (轻量级，本地文件存储)

### 2.4 安全护栏（已确定）

**多层防护策略**:

1. **输入层**: 敏感词过滤
   - 自杀/自伤关键词检测
   - 实现: 正则表达式 + 关键词库
   
2. **模型层**: 安全提示词
   - System Prompt中明确安全边界
   - 禁止医疗诊断、处方建议
   
3. **输出层**: 内容审核
   - 输出文本敏感词检测
   - 危机等级自动分类

**风险等级定义**:

| 等级 | 描述 | 处理方式 |
|------|------|----------|
| low | 正常对话 | 标准回复 |
| medium | 轻微负面情绪 | 加强共情，建议寻求支持 |
| high | 严重负面情绪/危机信号 | 触发handoff，提供求助热线 |

**实现位置**: `services/llm/safety/guardrails.py`

## 3. 输入与输出（必须对齐协议）

LLM 模块输入必须为 `PerceptionToLLM`，输出必须为 `LLMToDriver`（详见 [protocols.md](../protocols.md)）。

### 3.1 输入（PerceptionToLLM）

必选字段：

- `trace_id/session_id/turn_id`
- `constraints.deadline_ms/time_budget_ms`
- `asr.text`（或文本输入时同样填 `asr.text`）
- `memory.state.summary`（会话摘要）
- `memory.read.results[]`（检索到的记忆/偏好/事实）

推荐字段（提升共情与安全性）：

- `emotion.*`（融合后的情绪与置信度）
- `x_ext.fusion.psych_state`（融合模块的风险等级与筛查分数估计）
- `vision.summary` / `emotion.signals.voice.x_features`（用于语气与策略调节）

### 3.2 输出（LLMToDriver）

必选输出：

- `assistant.text`：对用户可读、可朗读的回复文本（短句、可分段）
- `actions[]`：至少包含 `tts_speak`

推荐输出：

- `render.voice`：语速/音高/情绪强度（适配陪护语气）
- `render.avatar`：表情/动作（与情绪一致）
- `actions: memory_write`：写入新事实/偏好/事件（支撑 10+ 轮记忆）
- `policy.safety`：风险分级与提示（用于驱动侧 UI 呈现或升级流程）

约束：

- 必须输出合法 JSON，不得混入额外文本
- 输出字段必须遵循 `protocols.md`，未知字段只能放在 `x_ext`

## 4. 上下文与对话记忆（≥10 轮，且不爆上下文）

### 4.1 记忆组成（已确定）

**三层记忆架构**:

```
┌─────────────────────────────────────────────────────────────┐
│                    记忆系统架构                              │
├─────────────────────────────────────────────────────────────┤
│  短期记忆 (Short-term)                                       │
│  - 最近6-10轮对话原文                                        │
│  - 存储: Redis (TTL: 1小时)                                  │
│  - 用途: 直接提供给LLM上下文                                 │
├─────────────────────────────────────────────────────────────┤
│  会话摘要 (Session Summary)                                  │
│  - 每轮更新的滚动摘要 (约200 tokens)                         │
│  - 存储: PostgreSQL + Redis缓存                              │
│  - 用途: 快速回顾会话主题                                    │
├─────────────────────────────────────────────────────────────┤
│  长期记忆 (Long-term)                                        │
│  - 结构化事实/偏好/事件                                      │
│  - 存储: PostgreSQL + ChromaDB向量检索                       │
│  - 用途: 跨会话记忆，个性化回复                              │
└─────────────────────────────────────────────────────────────┘
```

**记忆存储实现**:

| 记忆类型 | 存储介质 | 数据结构 | 实现位置 |
|----------|----------|----------|----------|
| 短期记忆 | Redis | List | `services/memory/short_term.py` |
| 会话摘要 | PostgreSQL | Text | `services/memory/session_store.py` |
| 长期记忆 | PG+ChromaDB | JSON+Vector | `services/memory/long_term.py` |

### 4.2 记忆写入（memory_write）

**写入流程**:

```
LLM输出记忆候选
       ↓
记忆服务验证 (去重/合并)
       ↓
隐私分级检查
       ↓
写入对应存储
       ↓
更新会话摘要
```

**写入类型定义**:

```json
{
  "memory_write": [
    {
      "type": "fact",
      "content": "用户最近失眠严重",
      "confidence": 0.85,
      "privacy": "long_term",
      "ttl_days": 30
    },
    {
      "type": "preference", 
      "content": "用户喜欢被称呼为'小李'",
      "confidence": 0.95,
      "privacy": "long_term",
      "ttl_days": 365
    },
    {
      "type": "episode",
      "content": "用户今天工作压力大，情绪低落",
      "confidence": 0.80,
      "privacy": "session",
      "ttl_days": 1
    }
  ]
}
```

**隐私分级**:

| 级别 | 说明 | 存储期限 | 示例 |
|------|------|----------|------|
| session | 会话级 | 当前会话 | 临时情绪状态 |
| long_term | 长期记忆 | 30-365天 | 偏好、重要事实 |
| private | 私密信息 | 需用户确认 | 敏感个人信息 |

## 5. 对话能力需求（陪护场景）

### 5.1 回复结构（建议统一范式）

每轮回复建议包含 3 段（按需裁剪）：

1. 共情与确认感受：承接用户情绪，避免直接给建议
2. 追问/语义澄清：引导补充背景（时间、诱因、强度、影响）
3. 小步建议/总结：给 1~2 个可执行的小步骤，并用一句话总结本轮重点

### 5.2 语义澄清能力

触发条件（示例）：

- ASR 置信度低或关键词不完整
- 用户表达含糊（“我不知道怎么办”、“很难受”）

行为要求：

- 先提出澄清问题，不做强结论
- 可复述用户可能的两种含义供选择（但避免模板化）

### 5.3 内容总结能力

触发条件（示例）：

- 连续 3~5 轮后自动生成短摘要（用于会话摘要更新）
- 用户主动要求“总结一下/你记住了吗”

输出要求：

- 总结必须可被 `memory_write` 结构化写入（facts/preference/episode）

## 6. 心理风险评估与专用能力（焦虑/抑郁/双相风险）

### 6.1 输入信号

本模块不直接“诊断”，只做“风险提示与策略分流”，输入信号来自：

- 融合模块输出：`x_ext.fusion.psych_state`（PHQ-9/GAD-7 估计分数、risk_level）
- 对话内容：用户自述的频率、持续时间、功能受损、自伤意念等线索
- 情绪强度与稳定性：`emotion.confidence`、多轮趋势（由记忆层或编排层提供）

### 6.2 输出策略

低风险：

- 以共情与问题澄清为主，给可执行小建议（睡眠卫生、呼吸练习、记录情绪等）

中风险：

- 更强调支持性对话、建议寻求现实支持（朋友/家人/校心理中心），并跟进是否有专业帮助渠道

高风险/危机（自伤他伤等）：

- 必须触发升级流程：`actions: handoff`（或在 `policy.safety` 标注高风险）
- 输出明确的安全建议与求助路径，避免提供危险细节或替代专业处置

## 7. 端到端时延治理（≤60s）

### 7.1 预算与超时

- 以 `constraints.deadline_ms` 贯穿推理
- 在 API 调用或自部署推理时设置：
  - 最大生成 token
  - 超时中断
  - 流式优先

### 7.2 降级策略（必须）

当接近超时：

- 输出“短共情 + 单一澄清问题”，并结束本轮
- 降低输出长度/禁用复杂检索/降低 top-k
- 切换更小模型或更快的 API 模型（若可用）

---

## 8. 可观测性与评测（建议）

必须记录（按 `trace_id` 关联）：

- LLM 总耗时、首 token 时间（若流式）
- 输入/输出 token 数（如可获得）
- 是否命中 RAG、top-k 命中文档数量
- 风险等级、是否触发 handoff

评测建议（对应 eval/latency、eval/memory）：

- 连贯性：10+ 轮对话不脱节（人工+脚本评测结合）
- 追问与澄清：在含糊输入场景的正确追问率
- 总结正确性：事实抽取的准确率与幻觉率
- 延迟：≤60s 的达标率（P95/P99）
